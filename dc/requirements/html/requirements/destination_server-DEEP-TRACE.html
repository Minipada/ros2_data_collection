    2. Plugin Loading
    Can load any number of destination plugins DS-PLU-001

    The Destination server must be able to load any number of destination plugins that conform to a specific interface. The plugins should be loaded dynamically at runtime.
    Can handle errors when loading destination plugins DS-PLU-002

    If the Destination server attempts to load a plugin that does not conform to the specified interface or the plugin file does not exist, it should throw a custom exception with a clear error message. The node should then stop processing.
    Extensibility and customizability DS-PLU-003

    The system should be easily extensible and customizable, allowing users to add new features, plugins, and functionality to meet their specific requirements in the repository.
    External extensibility and customizability DS-PLU-004

    The system should be easily extensible and customizable, allowing users to add new features, plugins, and functionality to meet their specific requirements in their own ROS packages.
    Can reload plugins without restarting the node DS-PLU-005

    The Destination server should be able to reload plugins dynamically without having to stop and restart the node. This will allow for updates and changes to be made to the plugins without interrupting data processing.
    Can unload plugins DS-PLU-006

    The Destination server should be able to unload plugins dynamically. This will allow for plugins to be removed from the system when they are no longer needed.
    Can provide information about loaded plugins DS-PLU-007

    The Destination server should be able to provide information about the plugins that are currently loaded, such as their names, version numbers, description and json schema.
    3. General
    Can batch samples before sending DS-GEN-001

    The destination system can batch multiple samples together before sending them to remote destinations, reducing network overhead and improving performance.
    Can store data locally if remote connection is not available DS-GEN-002

    If a remote connection is not available, the system should be able to store data locally and send it over the remote connection when it becomes available.
    Can store data locally if remote connection is slow or unreliable DS-GEN-003

    If a remote connection is slow or unreliable, the system should be able to store data locally and send it over the remote connection when it is stable.
    4. Backend
    Can store data using a backend MS-BAC-001

    The system should be able to store collected data using a backend, such as Fluent Bit or Redis, to handle backpressure and ensure data persistency.
    Support multiple backends MS-BAC-002

    The system should support multiple backends for storing collected data, handling backpressure, and ensuring data persistency. The selection of a backend should be configurable to allow for easy integration with different systems and tools.
    Can store data without a backend MS-BAC-003

    The system should be able to store collected data without a backend if needed, for example when testing or in environments where a backend is not available or practical to use. The system should provide a fallback option for data storage in such cases.
    Support for Fluent Bit MS-BAC-004

    The system should support the use of Fluent Bit as a backend to handle backpressure and ensure data persistency.

    Supports high throughput and low latency DS-PERF-001

    The system must support high throughput and low latency to handle large volumes of data and deliver near-real-time insights.
